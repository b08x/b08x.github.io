V2

Setting up an audio workstation can be a daunting task, even for experienced Linux users. It involves choosing a suitable distribution, fine-tuning the system for low-latency performance, configuring the workspace, and ensuring that audio devices work with Alsa, Jack, and Pulse. The list of tasks is lengthy.

This project aims to simplify the setup process by automating some of the tasks and managing configurations in a centralized location. In environments with multiple hosts, config changes can be pushed to multiple systems simultaneously, ensuring system parity. This is important for the concept of Open Music[1] as successful collaboration in this area relies on everyone working with the same code base.

Using Ansible allows for human-readable instructions to be written out as tasks in YAML, which is generally easier to make sense of as compared to Bash or Python. It allows for a bettering understanding of how the operating system is configured which makes troubleshooting future problems far less of a mystery.

Experimenting with and testing configurations, particularly in a networked environment, can be tedious and frustrating. Another aim of this project is to make that process less so. Jack periods and buffers are set set in vars/main.yml then running the playbook will set these paramters on the host as well as conduct latency and sound quality tests.[2] If the results are unsatisfactory, the values can be adjusted and the playbook re-run until the desired results are achieved. Once complete, the adjustments can be committed and pushed to a Git repository for future use and reference.

The reasons for taking this approach as opposing building an ISO is it allows for greater flexibility and quicker iteration when making changes to a system, especially within a networked environment. One design parameter of this project is have it scale to as many hosts as possible, which hopefully allows plenty of room for creative non-traditional setups. One host could be decidated to processing Spatial Audio, another for rendering, another for recording, another synths, etc etc.

This approach allows for more flexibility and quicker iteration when making changes to a system. It also saves time and resources by avoiding the need to rebuild ISOs and reinstall systems. Additionally, the ability to push changes ensures that all systems in the network are consistent and up-to-date, which is particularly important in a collaborative environment.
