Optimizing I3 Window Management for Automated Workspaces

The provided scenario outlines methods for saving and restoring i3 window layouts, which can be cumbersome for dynamic workflows. This analysis dives into optimizing this process, focusing on automation and efficiency for DevOps tasks.

Current Approach Limitations:

  • Manual Layout Restoration: Running scripts or using keybindings for each layout change disrupts workflow.
  • Static Configurations: .json files lack flexibility for dynamic environments where applications and their positions might change.
  • Verbosity: The i3-save-tree output requires manual editing, introducing potential for errors.

Recommended Solution: Dynamic and Automated Workspace Management

We can leverage i3's interfacing capabilities and scripting to achieve a more robust and flexible solution.

1. Centralized Workspace Configuration:

  • Shell Scripting: Create a central script (manage_workspaces.sh) to handle all workspace operations.
#!/bin/bash

# Function to save workspace layout
save_workspace() {
  local workspace_num=$1
  i3-save-tree --workspace "$workspace_num" |  # Pipe output for immediate processing
    sed 's|^\(\s*\)* "|\1"|g; /^\s*\/\*d' > ~/.config/i3/workspace_"${workspace_num}".json
}

# Function to restore workspace layout
restore_workspace() {
  local workspace_num=$1
  local layout_file="$HOME/.config/i3/workspace_${workspace_num}.json"

  # Check if layout file exists
  if [[ -f "$layout_file" ]]; then
    i3-msg "workspace $workspace_num; append_layout $layout_file"
  else
    echo "Error: Layout file not found: $layout_file"
  fi
}

# Example usage
save_workspace 2  # Save layout of workspace 2
restore_workspace 2 # Restore layout of workspace 2
  • Dynamic Application Launching: Instead of hardcoding application positions, include application launching within the script.
# ... (previous code from manage_workspaces.sh)

launch_workspace_apps() {
  local workspace_num=$1

  if [[ "$workspace_num" == "2" ]]; then
    # Example for workspace 2
    i3-msg "workspace $workspace_num"
    (uxterm &)
    (firefox https://github.com &)
  elif [[ "$workspace_num" == "3" ]]; then
    # Example for workspace 3
    i3-msg "workspace $workspace_num"
    (vim &)
    (docker-compose up -d &)
  fi
}

# Example usage
launch_workspace_apps 2

2. i3 Configuration Integration:

  • Keybindings: Bind script functions to specific key combinations for streamlined workflow.
# ~/.config/i3/config
bindsym $mod+Control+s exec "$HOME/.config/i3/manage_workspaces.sh save_workspace %w"
bindsym $mod+Control+r exec "$HOME/.config/i3/manage_workspaces.sh restore_workspace %w"

3. Advanced Automation (Optional):

  • Event-Driven Layouts: Trigger layout changes based on events like application launches. This involves tools like inotify to monitor file system events (e.g., new terminal opening) or dedicated i3 hooks.
  • Version Control: Manage your manage_workspaces.sh script with Git for better tracking and collaboration.

Advantages:

  • Automation: Eliminates manual file editing and command execution.
  • Dynamic Workspaces: Easily adapt to changing project needs by modifying the script.
  • Cleanliness: Removes the need for multiple .json files.
  • Maintainability: Centralized script simplifies updates and modifications.

Implementation Notes:

  • Adapt the provided script snippets to your specific applications and workflows.
  • Ensure the script has executable permissions (chmod +x manage_workspaces.sh).
  • For event-driven automation, refer to the i3 documentation and explore tools like inotify or incron.

By adopting this approach, you transition from static workspace configurations to a dynamic and automated system, significantly improving efficiency and maintainability for your DevOps workflow.

Using preset layouts in i3 :: Gideon Wolfe/

Save and restore the window layout

$ i3-save-tree --workspace N > ~/.config/i3/workspace_N.json

where N is the number of the preferred workspace. This will save the current layout of workspace N to the file ~/.config/i3/workspace_N.json.

The newly created file needs to be edited. This may be done automatically with the following command:
$ sed -i 's|^\(\s*\)* "|\1"|g; /^\s*\/\*d' ~/.config/i3/workspace_N.json

Restore the window layout of the workspace

There are two ways to restore the layout of the workspace: by writing a script, or by editing ~/.config/i3/config to automatically load the layout.

To restore the saved layout in the previous section, write a file named load_layout.sh with the following contents:

The starting lines:

~/load_layout.sh

#!/bin/sh
i3-msg "workspace M; append_layout ~/.config/i3/workspace_N.json"

where M is the number of the workspace in which you would like to load the previously saved layout and N is the number of workspaces saved in the previous section.

And the commands used in the previous section to get the preferred windows, but enclosed in parentheses and with an ampersand appended before the last parentheses.
For example, if the saved layout contained three uxterm windows:

~/load_layout.sh

#!/bin/sh

## First We Append the Saved Layout of Workspace N to Workspace M

i3-msg "workspace M; append_layout ~/.config/i3/workspace_N.json"

## And Finally We Fill the Containers with the Programs They Had

(uxterm &)
(uxterm &)
(uxterm &)

Then set the file as executable. Finally, the layout of workspace N can be loaded onto to workspace M by running:

$ ~/load_layout.sh

Tip: Adding bindsym $mod+g exec ~/load_layout.sh to ~/.config/i3/config and restarting i3 will bind Mod+g to run the above script.

Note: If the above script does not work properly, refer to the official documentation. The swallows sections of ~/.config/i3/workspace_N.json needs to be manually edited.

4.1. Restoring a vertically split workspace

When using i3-save-tree with the --workspace switch, only the contents of the workspace will be dumped. This means that properties of the workspace itself will be lost.

This is relevant for, e.g., a vertically split container as the base container of a workspace. Since the split mode is a property of the workspace, it will not be stored. In this case, you will have to manually wrap your layout in such a container:

// vim:ts=4:sw=4:et
{
	// this is a manually added container to restore the vertical split
	"layout": "splitv",
	"percent": 0.5,
	"type": "con",
	"nodes": [

		// the dumped workspace layout goes here

	]
}
