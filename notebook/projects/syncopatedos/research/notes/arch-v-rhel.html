Arch Vs debian|fedora

with arch, packages are basically scripts that download|clone the source, builds it, and installs the upstream source

with debian|fedora, packages have an additional layer of configuration designed to unify upstream and "stabilize" source installs in some way.

In terms of stability, while Debian chooses to enforce a sensible policy, once the user becomes involved with the system, the structured software release cycle really only benefits highly structure enterprise environments, in this case especially as people work with sound like to experiment. If Debian is using a stock gcc version that doesn't allow for compiling something that requires a newer version, we'll...you could say "well setup a testing environment in your studio" to which I would reply "ok. please diagram that architecture as their currently is no standardized process specifically for testing audio applications".

Potential Instability: Due to its rolling release model, there's a chance of encountering occasional instability or compatibility issues after updates.

Throughout my personal experience involved with Linux systems, one of the more obnoxious system failures was about 6 years when Debian packaging changed how labeled mysql packages, which I was only made aware of after running an update that broke our backend. About

This project originally started out as a customized Linux distribution for Audio Production.

as an Ansible role to manage a single Linux workstation designated for Audio Production.

Most of the work was testing or experimenting with different software and OS configurations, it made sense to start using Ansible to manage the host and application configurations

When tests or experiments were successful, the working configs then committed a central repository.

Once this was working well, it had occurred to me to start pushing the idea of using Ansible to manage a "Linux-Audio" workstation. The problem with this idea however, is the steep learning curve involved, especially with Ansible.

Over years of research and development , one complaint I see the most often about "Linux-Audio" is that musicians spend more time with configuration and troubleshooting than they do actually making music. So then the objective became to structure this project in way that doesn't add another layer of complexity to the Musician/Engineer workflow.

Once that objective was set, it started to make sense to frame this project as an exercise in setting up a small audio production studio/live environment; deploy and configure DAW workstations, networked audio, pi synth nodes, pi mic arrays, etc, etc

It's about Real Mother Fucking time

If an audio processing system using Linux needs to
provide the performance of professional audio quality, the round trip latency of audio samples has to be as minimal (some of the pro audio tools have round-trip latency as minimal as 1 ms [14]) and as per professional musicians observation audibly noticeable latency is anything above 12 ms [14]. Hence, a customized Linux kernel
is needed which can schedule the real-time audio processing task with the highest
priority and provide deterministic worst case latency which meets the requirement.

2.1 Real Time Systems
A real time system, by definition, is that which reacts to stimuli within a finite
and predictable amount of time [11]. Proper functionality of such systems not only
require logical correctness of the output, but also, the amount of physical time
within which these outputs are produced [28]. For example as shown in Fig. 2.1
an audio processing system is a real-time system where input samples act as the
stimulus and must be processed to produced as output signal within certain amount
of time. Failing to produce the output signal within the required time, the system
is considered as functionally improper.

Audio processing - Software architecture

Fig. 2.5 shows a generic software architecture followed by many operating systems to handle audio services.

Audio services are basically tasks such as playback from
files, recording to files or streaming of audio from input device to output device.

They can be divided into four layers, the top one being the application layer and the bottom one being the hardware device which is usually the sound card.

As with many of the general purpose operating systems the device driver in the kernel acts as an interface between the user space applications and the hardware

The flow of actions performed by the operating system to process data for streaming applications is explained
briefly below:

  1. The hardware device (sound card) informs the processor through an hardware interrupt when it has audio samples ready to be processed.
  2. The vkernel driver waits for the interrupt request and captures the audio samples from the hardware deice. In addition to capturing, it also sends the audio samples which are to be sent to the output channel. Since a chunk of audio samples are handled at a time, there are two buffers in the kernel driver to store the audio samples.

####The Capture buffer== is a block of memory allocated to store input audio samples

####Output buffer is used to store the processed samples which will be sent out.==

The kernel driver uses DMA for data transfer, thus allowing to do computation and communication of data in
parallel.

  1. After initiating the data transfer using a DMA controller, the kernel driver wakes up the user space application waiting for audio samples through a call-back function at every interrupt.

  2. User space programs can use the driver services only through system calls such as open(), ioctl(), etc. A user space library provides an abstraction for these system calls and provides easier interfaces for application developers. This library provides the callback function of processing a block

This library provides the callback function of processing a block of audio data at every interrupt. The call back function has two arguments which are pointers to the input and output buffers as shown in the figure 2.5.

  1. At every callback from the kernel driver, the application is woken up and processes the input buffer and puts it into the output buffer.

Buffer size: The Buffer size is the size of both the Capture buffer and the Output buffer. Hence, it is also the size of the block of audio data that will be processed at every interrupt.

Interrupt period: The Interrupt period is the average time duration between two successive interrupts from the audio hardware device. The Interrupt period is dependent on the Buffer size. If larger buffer sizes are used then a large chunk is processed at each interrupt. Thus, lesser number of interrupts are needed in a given period of time.

Round-trip audio latency: Round-trip audio latency refers to the delay period between when an analog audio signal enters and when it emerges from a system

Configuring Sound Card Load order

Fedora speeds up the boot process by loading device drivers for hardware in parallel. That means the load order of the drivers for soundcards is not predictable. If you are not using USB soundcards there is a possible workaround (see below), otherwise it is impossible to make sure the cards have consistent fixed indexes as USB drivers are loaded very early in the boot process.

In any case, it is now recommended to address soundcards by name instead of index number (which can change from boot to boot).

### **Configuration**

There is almost nothing to configure if you are using the Planet CCRMA version of hotplug and ezusbmidi.

1. **add usb-midi and audio to the /etc/hotplug/blacklist file** So that the OSS audio and usb-midi modules are not automatically loaded when the device reconnects after the firmware download. Add ``usb-midi'' and ``audio'' in separate lines at the end of the list of blacklisted modules in that file.
    
2. **make sure you have configuration ``space'' for the soundcard** Make sure that in your /etc/modules.conf the Options line for snd has card_limit=3 if you have one soundcard, or 4 if you have two (the current alsa version in Planet CCRMA uses two devices for the Midisport). Otherwise the snd-usb-audio and snd-usb-midi modules will not be able to be loaded on demand.
    

So, su root, do a ``tail -f /var/log/messages'' to see what happens and insert the Midisport 1x1 or 2x2 to a working USB interface. If all LEDs eventually blink once, then turn off and the USB LED blinks very fast then you should be in bussiness.

If you do a ``cat /proc/asound/cards'' you should see two cards in addition to your normal audio card entry (or entries). The first card is a ``dummy'' usb audio card with no devices, the second card is the usb midi device and should contain one or two midi devices, depending on which interface box you have. Do a ``cat /proc/asound/devices'' to see what you have.

 I could use **hw:0** but the index could change in the next boot. The answer is to use **hw:M66** instead (the name is in between square brackets). If you do that you will always use the proper card regardless of the load order of device drivers.


`/sbin/lsmod | grep ^snd`

This will list all the soundcard related kernel modules. From the list and the output for ``cat /proc/asound/cards'' you should be able to figure out which is which.

Here is a filled in example that includes three cards:

alias snd-card-0 snd-ice1712
options snd-card-0 index=0
options snd-ice1712 index=0
alias snd-card-1 snd-ens1370
options snd-card-1 index=1
options snd-ens1370 index=1
alias snd-card-2 snd-usb-audio
options snd-card-2 index=2
options snd-usb-audio index=2
