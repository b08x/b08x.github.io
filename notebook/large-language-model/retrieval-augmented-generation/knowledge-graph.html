Knowledge Graph Construction from Text

Quote
  • Knowledge graphs compress the vast diversity and redundancy of strings into unique concepts. Distinguishing nodes in knowledge graphs doesn't depend on spelling but on meaning – so nodes can't be simple strings. If there are distinct nodes in a graph for dog, pooch, mutt, stray, mongrel, hound, cur, man's best friend, and others, then it does not model unique concepts. Clustering strings is at best a crude approximation to this. So graphs with strings as nodes aren't knowledge graphs.
  • Knowledge graph nodes enable unpacking a concept to display and leverage information about the components and characteristics of things in the world. Nodes in knowledge graphs are pointers, not literal values like strings, so they point to and document meaning for our algorithms. Clusters and embeddings of strings like 'elephant' include no information about weights, heights, trunks, and tusks – only about other strings in sentence context.
  • Knowledge graph nodes are unique, not ad hoc agglomerations like Actors and Directors or Influencers and Entrepreneurs. Clusters of strings are notoriously difficult for humans to label and evaluate because the criteria for clustering (other strings in context) are so different from what humans use to delimit concepts (components and characteristics). Clusters of strings, then, only sometimes and approximately correspond to the categories that humans define and manipulate to build knowledge.
  • Knowledge graphs have varied types of relations. Hierarchical clustering only yields one single, fuzzy notion of "relatedness" between strings, one type of relation – an important but very small subset of what goes into a knowledge graph. Rich knowledge graphs require relations of different types between categories: partOf, usedFor, hasColor, etc. to document a range of facts about things in the world, as well as relations between individuals like spouseOf or createdBy. Clustering provides none of these.
  • Clustering and embeddings do no more than situate strings as somehow related to other strings. Knowledge graphs enable algorithms to interpret strings, i.e., to map strings to conceptualizations of things in the world.
    source

1. Summary

This scenario highlights the limitations of using string clustering and embeddings for building knowledge graphs. While these techniques can identify relationships between words based on co-occurrence, they fall short of capturing the rich semantic relationships and conceptual understanding needed for a true knowledge graph.

The challenge lies in moving beyond simple string representations to a graph structure where nodes represent unique concepts and edges represent various relationships between them.

2. Proposed Solution

Building a knowledge graph from text is a complex task involving several steps. Here's a simplified approach using Ruby and existing NLP libraries:

2.1 Named Entity Recognition (NER) and Entity Linking:

This is indeed a crucial first step. NER helps identify and classify entities within the text, while entity linking connects these mentions to real-world entities in knowledge bases.

Here's an improved Ruby code example using the ruby-spacy gem:

require 'spacy'

# Load the English language model with entity linking capabilities
nlp = Spacy::Language.load('en_core_web_lg')

text = "Barack Obama was the 44th president of the United States."

# Process the text
doc = nlp.read(text)

# Extract entities and their Wikidata IDs
entities = doc.ents.map do |ent| 
  { 
    entity: ent.text, 
    type: ent.label_,
    wikidata_id: ent.kb_id # Access Wikidata ID if available
  } 
end

puts entities
# Expected Output (format may vary): 
# [{:entity=>"Barack Obama", :type=>"PERSON", :wikidata_id=>"Q76"}, 
#  {:entity=>"44th", :type=>"ORDINAL", :wikidata_id=>nil}, 
#  {:entity=>"United States", :type=>"GPE", :wikidata_id=>"Q30"}]

2.2 Relationship Extraction

The next critical step is relationship extraction. This involves identifying how the extracted entities are related within the text.

  • Tools: We can leverage SpaCy's dependency parsing capabilities for this task.
  • Code Example:
require 'spacy'

nlp = Spacy::Language.load('en_core_web_lg')

text = "Barack Obama was the 44th president of the United States."
doc = nlp.read(text)

# Assuming 'entities' array from the previous step
entities.each do |entity|
  token = doc.find { |t| t.text == entity[:entity] } 

  if token
    head_token = token.head
    entity[:relation] = head_token.text
    entity[:head_entity] = entities.find { |e| e[:entity] == head_token.text }
  end
end

puts entities

2.2 Relation Extraction:

  • Goal: Identify relationships between entities extracted in the previous step.
  • Tools: Open Information Extraction (OpenIE) libraries like openie-standalone can extract relations expressed in text.
  • Code Example:
require 'openie-standalone'

client = OpenIE::Client.new

text = "Apple Inc. is headquartered in Cupertino, California."

relations = client.relations(text)

puts relations
# Example Output:
# [
#   {:subject=>"Apple Inc.", :relation=>"is headquartered in", :object=>"Cupertino, California"}
# ]

2.3 Knowledge Graph Construction

  • Goal: Represent the extracted information in a graph structure.
  • Tools: Consider using graph databases like Neo4j or libraries like RDF.rb for RDF representation.
  • Example (Conceptual using RDF.rb):
require 'rdf'
require 'rdf/turtle'

# ... (previous code for NER and Relationship Extraction)

# Create an RDF graph
graph = RDF::Graph.new

entities.each do |entity|
  subject = RDF::URI.new("http://example.org/entity/#{entity[:entity]}")
  graph << [subject, RDF.type, RDF::URI.new("http://schema.org/#{entity[:type]}")]

  if entity[:head_entity]
    object = RDF::URI.new("http://example.org/entity/#{entity[:head_entity][:entity]}")
    graph << [subject, RDF::URI.new("http://example.org/relation/#{entity[:relation]}"), object] 
  end
end

puts graph.dump(:turtle)

2.3 Knowledge Graph Population:

  • Goal: Create nodes and relationships in your knowledge graph based on the extracted information.
  • Tools: Use graph databases like Neo4j or RDF stores like GraphDB. Ruby drivers are available for both.
  • Code Example (Neo4j):
require 'neo4j'

neo4j_session = Neo4j::Session.open(uri: 'bolt://localhost:7687', basic_auth: { username: 'neo4j', password: 'your_password' })

relations.each do |relation|
  neo4j_session.query("MERGE (s:Entity { name: '#{relation[:subject]}' })")
  neo4j_session.query("MERGE (o:Entity { name: '#{relation[:object]}' })")
  neo4j_session.query("MERGE (s)-[:#{relation[:relation].gsub(/\s+/, '_')}]->(o)")
end

3. Troubleshooting

  • Entity Linking Accuracy: Entity linking can be challenging. Ensure you're using a model with good linking capabilities and consider fine-tuning on a domain-specific dataset if needed.

  • Ambiguity: Natural language is inherently ambiguous. Implement disambiguation techniques and potentially incorporate rule-based systems to improve accuracy.

  • Scalability: Building knowledge graphs from large datasets can be computationally intensive. Explore efficient graph database solutions and optimization techniques.

4. Best Practices

  • *Start Small: Begin with a well-defined domain and gradually expand your knowledge graph.

  • Contextual Embeddings: While this scenario focuses on moving beyond embeddings, consider incorporating contextual embeddings like BERT to enhance entity and relation extraction.

  • Iterative Development: Building a knowledge graph is an iterative process. Start with a small, well-defined scope and gradually expand.

  • Visualization: Visualize your knowledge graph to aid understanding and analysis.

5. Additional Considerations

  • External Knowledge Bases: Leverage existing knowledge bases (DBpedia, Wikidata) to enrich your knowledge graph with additional facts and relationships.

  • Ontology Engineering: For complex domains, consider formal ontology engineering to define concepts and relationships more rigorously.

  • *Reasoning: Implement reasoning mechanisms to infer new knowledge from existing facts in your graph.

A graph database is a type of NoSQL database that uses graph structures to store and query data. It consists of nodes (also called vertices) and edges (also called relationships) that connect those nodes. Nodes represent entities, while edges represent the relationships between those entities. Graph databases are optimized for traversing and querying complex relationships, making them an excellent choice for applications that require high levels of interconnectedness.

Cypher is a declarative graph query language designed specifically for querying and manipulating graph data

Example Class Recursive Knowledge Graph Query

class RecursiveKnowledgeGraphQuery
  attr_reader :knowledge_graph, :llm

  def initialize(knowledge_graph:, llm:)
    #knowledge_graph = knowledge_graph
    #llm = llm
  end

  def query(initial_query)
    query_with_context(initial_query, [])
  end

  private

  def query_with_context(query, context)
    extracted_info = llm.extract_information(query, context)
    knowledge_graph.add_information(extracted_info)

    full_context = knowledge_graph.retrieve_context(query)
    answer = llm.query_graph(query, full_context)

    if answer.sufficient?
      return answer
    else
      query_with_context(answer.missing_context_query, full_context)
    end
  end
end
