Document Processing Workflow using Ruby Draft 1

Overview

Here is a high level summary of the process

First we index our knowledge base. We load the individual documents using a loader, and then use a splitter to turn it into bite-size chunks or snippets. Then LDA Topic Modeling is applied to those snippets. Parent documents are assigned the top 3 occurring topics totaled up after all of the document snippets have been processed.

Using rubygem ohm we can store the topics as a separate model in Redis, with all of the words found for each topic. So if the topic is "audio engineering", part of the query retrieval will include all stored words associated with that topic.

semi-relational-data-modeling-redis-ohm

Once we have snippets with topics, we pass them to our embedding function, which turns them into vectors that can be used for semantic searching. We save these embeddings, alongside their text snippets in our vector database.

Next comes retrieval. It starts with the question, which is then sent through the same embedding fucntion and passed into our vector database to determine the closest matched snippets, which we’ll use to answer the question.

The approach involves converting the user's input into a sequence of intermediate symbolic memory operation steps, enabling LLMs to more accurately and effectively determine the most accurate context in which to respond to.

ChatDB: Augmenting LLMs with Databases as Their Symbolic Memory/
https://arxiv.org/pdf/2306.03901.pdf

Finally, augmented answer generation using chain of memory. We take the snippets of knowledge, format them alongside a custom system prompt and our question, and, finally, get our context-specific answer.

The key steps are:

  1. User input is classified by LLM to determine intent
  2. Lookup cache in Redis searches for cached prompts based on intent
  3. ChromaDB performs semantic search using vectorized input to find relevant passages
  4. Cached prompts and relevant passages are combined to construct the final prompt
  5. Final prompt is passed to LLM text generator to create response
  6. Generated response is returned to the user

And that's about it.

Chunking Methods

If chunking by sentence is not possible or not suitable for your specific use case, there are alternative approaches you can consider for organizing and processing text data. Here are a few options:

  1. Fixed-size Chunks Divided into Clusters of related content:

    • Divide the text into fixed-size chunks, such as by a certain number of characters or words. This way, each chunk has a consistent size, regardless of sentence boundaries. Then Use topic modeling algorithms to identify clusters of related content and treat each cluster as a chunk. Redis required

  2. Paragraph-based Chunks:

    • Use natural language processing (NLP) techniques to segment the text into sentences. Assuming that a paragraph consists of one or more sentences, this approach can help identify paragraphs.
    • Then Use topic modeling algorithms to assign a topic to the paragraph.
    • create all of the words in the paragraph in redis, referencing the chunk_id

  3. Token-based Chunks:

    • Divide the text based on tokens, where tokens can be words, phrases, or any defined unit of text.
    • Example: Divide the text at every punctuation mark, treating each sentence or phrase as a chunk.

  4. Semantic or Topic-based Chunks:

    • Use natural language processing techniques to identify and group text related to specific topics or themes.
    • Example: Use topic modeling algorithms to identify clusters of related content and treat each cluster as a chunk.

  5. Hybrid Approaches:

    • Combine multiple approaches based on different criteria, such as using a combination of fixed-size chunks and sentence-based chunks.
    • Example: Divide the text into fixed-size chunks and further divide those chunks based on sentence boundaries.

The choice of chunking method will depend on the nature of your text data, the requirements of your application, and the specific analysis or processing you intend to perform on the chunks. Experimenting with different approaches and evaluating their effectiveness for your use case is key to finding the most suitable chunking strategy.

Step 4: Apply Topic Modeling Using LDA
Tomoto

This gem will be used for topic modeling using Latent Dirichlet Allocation (LDA) technique. It will analyze the text and identify the underlying topics within the document collection.

The goal of LDA is to infer the topic distribution for each document and the word distribution for each topic.
To perform topic modeling using LDA (Latent Dirichlet Allocation), we will use the tomoto RubyGem.

LDA is a generative statistical model that allows us to discover hidden topics within a collection of documents. It assumes that each document is a mixture of various topics, and each word in the document is generated from one of the topics. The goal of LDA is to infer the topic distribution for each document and the word distribution for each topic.

Here's an example code snippet that demonstrates how to perform topic modeling using LDA with the tomoto RubyGem:

require 'tomoto'

# Load the corpus
corpus = [
  "This is the first document",
  "This document is the second document",
  "And this is the third one",
  "Is this the first document"
]

# Create a LDA model
model = Tomoto::LDA.new

# Preprocess the corpus
corpus.each do |doc|
  tokens = doc.split
  model.add_doc(tokens)
end

# Train the model
model.train(100) # Specify the number of iterations

# Get the topic distribution for each document
corpus.each_with_index do |doc, idx|
  topic_dist = model.docs[idx].topic_dist
  puts "Document #{idx + 1}: #{topic_dist}"
end

# Get the word distribution for each topic
model.each_topic do |topic|
  word_dist = topic.word_dist
  puts "Topic #{topic.id}: #{word_dist}"
end

In this code snippet, we first load the corpus, which is an array of documents. Each document is represented as a string.

Next, we create an instance of the Tomoto::LDA class, which represents the LDA model. We then preprocess the corpus by splitting each document into tokens and adding them to the model using the add_doc method.

After preprocessing, we train the model by calling the train method and specifying the number of iterations. The more iterations we specify, the better the model will be trained, but it will also take longer to train.

Once the model is trained, we can access the topic distribution for each document by iterating over the documents and calling the topic_dist method on each document.

Similarly, we can access the word distribution for each topic by iterating over the topics using the each_topic method and calling the word_dist method on each topic.

Note: This code assumes that you have already installed the tomoto RubyGem. You can install it by running gem install tomoto in your terminal.

require 'tomoto'

def apply_topic_modeling(documents)
  # Create a new LDA model
  model = Tomoto::LDA.new

  # Preprocess the documents and add them to the model
  documents.each do |document|
    model.add_doc(document)
  end

  # Train the model
  model.train(100) # Specify the number of iterations for training

  # Get the topics and their word distributions
  topics = []
  model.k.times do |topic_id|
    topic = {
      id: topic_id,
      words: model.get_topic_words(topic_id)
    }
    topics << topic
  end

  topics
end

In this function, we first create a new LDA model using Tomoto::LDA.new. We then preprocess each document in the input list and add it to the model using model.add_doc(document). After that, we train the model for a specified number of iterations using model.train(100).

Finally, we retrieve the topics and their word distributions using model.get_topic_words(topic_id). We iterate over the topics and create a hash for each topic, including its ID and word distribution. We store these topic hashes in the topics array, which is returned as the result of the function.

Using the Ruby Gem ohm to Query and Retrieve Document Information from Redis

In this example, we have Ohm models related to NLP document processing:

The provided Ruby file defines the structure for organizing and processing various components of textual information. Let's break down what each part does:

  1. Collection:

    • Represents a collection of documents and topics.
    • It has a name and contains documents and topics.

  2. Document:

    • Represents a single document.
    • Contains a title, content, information about whether it has been processed or not, and a collection of chunks.
    • Belongs to a topic and a parent collection.
    • Provides a method to process the document.

  3. Chunk:

    • Represents a chunk of text within a document.
    • Contains text-related attributes such as tokenized text, sanitized text, and a collection of words.
    • Belongs to a document and a topic.
    • May have associated vector data.

  4. Word:

    • Represents a word in a chunk of text.
    • Contains attributes like the word itself, synsets, part of speech, and named entity.
    • Belongs to a chunk.
    • May have associated vector data.

  5. Topic:

    • Represents a topic related to the documents and chunks.
    • Contains a name, description, and vector information.
    • Has collections of documents and chunks associated with it.
    • Belongs to a parent collection.

  6. Vector Data:

    • Represents vector data associated with either a chunk or a topic.

This structure allows for the organization and manipulation of textual data, facilitating tasks like document processing and topic analysis. Each component plays a crucial role in managing and understanding text in a structured and efficient manner.

These Ohm::models allow for representing and associating collections, documents, chunks, words, topics, and vector data in the context of NLP document processing.

require 'ohm'
require 'ohm/contrib'

# Collection object_bucket
class Collection < Ohm::Model
  attribute :name
  collection :documents, :Document
  collection :topics, :Topic
  unique :name
  index :name
end

# Document object_bucket
class Document < Ohm::Model
  attribute :title
  attribute :content
  attribute :processed, Ohm::DataTypes::Boolean
  collection :chunks, :Chunk
  reference :topic, :Topic
  reference :collection, :Collection  # Reference to the parent collection
  unique :title
  index :title
  index :processed

  def process!
    # Perform document processing here
    self.processed = true
    save
  end
end

# Chunk object_bucket
class Chunk < Ohm::Model
  attribute :text
  attribute :tokenized_text
  attribute :sanitized_text
  collection :words, :Word
  reference :document, :Document
  reference :topic, :Topic
  list :vector_data, :VectorData
end

# sentence?

# Word object_bucket
class Word < Ohm::Model
  attribute :word
  attribute :synsets
  attribute :part_of_speech
  attribute :named_entity
  reference :chunk, :Chunk
  collection :vector_data, :VectorData
end

# Topic object_bucket
class Topic < Ohm::Model
  attribute :name
  attribute :description
  attribute :vector
  collection :documents, :Document
  collection :chunks, :Chunk
  reference :collection, :Collection  # Reference to the parent collection
  unique :name
  index :name
end

# Vector Data object_bucket
class VectorData < Ohm::Model
  attribute :vector
  reference :chunk, :Chunk
  reference :topic, :Topic
end

Step 8: Vector Database for Semantic Search

Vectorizing Words

Okay, with that context the dataset being scientific papers and documentation related to Linux audio, here are my thoughts:

  • Vectorizing words first is likely a good starting point. This will allow you to create useful embeddings for key technical words and concepts that occur frequently in this domain.

  • Then you can move up to vectorizing sentences, paragraphs, sections etc. This will capture more context and meaning.

  • Specifically for a troubleshooting assistant, capturing sentence/paragraph level semantics will be important to understand full technical descriptions of issues, solutions, configurations etc.

  • Vectorizing some larger chunks like entire papers/sections may be less useful for this application vs. focusing on smaller units of text.

  • You want to balance having good word-level representations but also sufficient context from larger chunks. Sentences and paragraphs are probably a sweet spot.

  • One idea is a hierarchical approach: word vectors -> sentence vectors -> paragraph vectors. This builds up from words to capture meaning at different levels.

So in summary, starting with quality word vectors, then combining them into sentence/paragraph representations would be my recommended approach for your dataset and use case. Let me know if you have any other questions!

Averaging the Word Vectors in a Sentence to Get a Sentence Vector

require 'numo/narray'  # You can use Numo::NArray for efficient numerical operations

# Sample word vectors (replace with your actual word vectors)
word_vectors = {
  'apple' => [0.1, 0.2, 0.3],
  'banana' => [0.4, 0.5, 0.6],
  # Add more word vectors as needed
}

# Sample sentence
sentence = ['apple', 'banana']

# Function to calculate the sentence vector by averaging word vectors
def calculate_sentence_vector(sentence, word_vectors)
  # Initialize an array to store word vectors
  vectors = []

  # Iterate over words in the sentence and get their vectors
  sentence.each do |word|
    vector = word_vectors[word]
    vectors << vector if vector
  end

  # Convert vectors to Numo::NArray for efficient numerical operations
  vectors = Numo::NArray[*vectors]

  # Calculate the mean along the rows to get the sentence vector
  sentence_vector = vectors.mean(axis: 0)

  sentence_vector.to_a  # Convert the Numo::NArray to a regular Ruby array
end

# Calculate the sentence vector
sentence_vector = calculate_sentence_vector(sentence, word_vectors)

# Print the sentence vector
puts 'Sentence Vector:', sentence_vector.inspect
